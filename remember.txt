History of Changes - Gemini Integration Fixes

1. INVESTIGATING 404 NOT_FOUND ERROR
- What: Investigated `test_gemini.py` failing with a 404 error on `gemini-1.5-flash`.
- Why: The provided API key/account did not have access to or no longer supported that specific model.
- How: Listed available models via Python SDK and updated `test_gemini.py` and `chat_engine.py` to use `gemini-2.0-flash`.

2. INVESTIGATING 429 BURST ERRORS (RATE LIMITING)
- What: Discovered 40+ `429 TooManyRequests` errors appearing on the Google AI Studio dashboard simply from running a test script.
- Why: Google's Free Tier strictly enforces 15 requests-per-minute. The `langchain-google-genai` wrapper had an aggressive automatic retry mechanism; when the script hit the limit, Langchain rapidly retried dozens of times in the background.
- How: Added `max_retries=1` to the `ChatGoogleGenerativeAI` constructor in `test_gemini.py` to fail fast, and `max_retries=2` in `chat_engine.py` to prevent hanging. Also added explicit 429 quota exception handling in `main.py`.

3. BYPASSING THE LOCKED QUOTA
- What: The `gemini-2.0-flash` 15-RPM quota bucket was continuously maxed out (locked) by background processes or past loops.
- Why: We needed a way to test and use the API immediately without being rate-limited.
- How: Discovered through testing that newer models have separate quota buckets, so updated `test_gemini.py` and `chat_engine.py` to use `gemini-2.5-flash` instead.

4. MIGRATING TO GEMINI-3-FLASH-PREVIEW
- What: The user reviewed documentation showing `gemini-3-flash-preview` and wanted to try it.
- Why: To use the newest experimental model and enjoy a completely fresh quota bucket.
- How: 
   - Replaced `gemini-2.5-flash` with `gemini-3-flash-preview` in both scripts.
   - Discovered the new 'gemini-3' model returns its response as a list of dictionaries (parts) instead of a plain string.
   - Updated the print statement in `test_gemini.py` to cleanly extract actual text (`"".join(part.get("text", "")...)`) so it displays properly in the terminal.

9. ARCHITECTURE OPTIMIZATION: FASTAPI STARTUP

14. BUG FIX: CHAT RETURNING WRONG REPO DATA
- What: The `/chat` and `/search` endpoints were returning results from ALL repositories instead of just the one being analyzed.
- Why: `search_code()` in `vector_store.py` performed a global pgvector similarity search without filtering by `repo_name`. When multiple repos were embedded, the AI would answer with context from the wrong codebase.
- How: Added `repo_name` parameter to `search_code()`, passed it through from `chat_engine.py`, and implemented a dual strategy: tries Supabase RPC with `filter_repo_name` first, falls back to global search + client-side Python filtering if the RPC function doesn't support the parameter yet.

16. DIAGRAM VISIBILITY IMPROVEMENT
- What: Rewrote `diagram_generator.py` to produce cleaner, more readable Mermaid charts.
- Why: Node labels used full file paths (e.g., `backend/app/chat_engine.py`) making every node massive and the entire diagram compressed into unreadable tiny boxes.
- How:
  - Added `shorten_label()` function that extracts just the filename (e.g., `chat_engine.py`) for display labels while keeping full paths as unique node IDs.
  - Changed layout direction from `graph TD` (top-down) to `graph LR` (left-to-right) for better horizontal use of the canvas.
  - Added edge deduplication to prevent duplicate connections from cluttering the chart.

17. CHAT SEARCH 2-STRATEGY REWRITE
- What: Completely rewrote `vector_store.py` search_code() with a robust 2-strategy approach, and updated `chat_engine.py` to always pass repo context to the LLM.
- Why: The Supabase RPC `match_code_chunks` function doesn't return a `repo_name` column in its results. Previous code tried to filter results by repo_name, but since the field didn't exist in the response, it returned unfiltered global results (wrong repo data) or empty results.
- How:
  - Strategy 1: RPC vector search — checks if results actually contain a `repo_name` field. If they do, filters by it. If they don't, falls through to Strategy 2 instead of returning wrong-repo data.
  - Strategy 2 (fallback): Direct Supabase table query with `.eq("repo_name", repo_name)` — guarantees results from the correct repository.
  - Updated `chat_engine.py` to pass `repo_name` into the LLM prompt template and provide a fallback message when no embeddings exist.

18. FRONTEND-BACKEND FULL WIRING (handleIngest)
- What: Updated `LandingView.jsx` with a `handleIngest` function that chains two API calls before transitioning to the dashboard.
- Why: The original `handleSubmit` only called `/ingest`. The user needed `/embed` to also run automatically so the chat has context when the dashboard loads.
- How: 
  - Step 1: `POST /ingest` — clones the repo. Button shows "Cloning repository..."
  - Step 2: `POST /embed/{repo_name}` — generates vector embeddings. Button shows "Generating embeddings..."
  - On success: calls `onAnalyze(repoName)` to transition to the Dashboard view.
  - Dynamic `statusMsg` state updates the button text at each phase so the user always knows what's happening.


19. UVICORN RELOAD CRASH FIX
- What: Renamed temp_repos to .temp_repos across the backend.
- Why: Uvicorn\'s watchfiles on Windows ignored the --reload-exclude flag, constantly rebooting the server during the clone process and causing 500 Internal Server Errors.
- How: Hidden directories (starting with a dot) are natively ignored by watchfiles. Updated git_service.py and all routes in main.py to use .temp_repos.

20. SHALLOW CLONE OPTIMIZATION
- What: Added depth=1 to the GitPython clone command.
- Why: Large repositories (like python-mini-projects) took 5+ minutes to clone because it downloaded the entire commit history, causing frontend HTTP requests to time out.
- How: Repo.clone_from(github_url, target_dir, depth=1) massively reduces download times to seconds.

21. MERMAID DIAGRAM RENDER FIXES
- What: Fixed diagram_generator.py to prevent blank canvases or raw-text crashes.
- Why: Repositories with no imports (or non-Python repos) produced empty graphs (graph LR;), crashing the mermaid.js parser. Isolated Python files also failed to show up.
- How: Refactored generator to explicitly declare all nodes before edges. Added a fallback node if no .py files are found, ensuring the UI always renders a valid SVG.

22. UI ERROR SANITIZATION
- What: Added sanitize_error wrapper around all FastAPI exceptions in main.py.
- Why: When Supabase went down or the URL was invalid, Python\'s Supabase SDK threw an exception containing a massive raw Cloudflare HTML page. The frontend rendered this raw HTML, ruining the UI layout.
- How: sanitize_error intercepts 521 Cloudflare and getaddrinfo errors, replacing them with a single clean sentence. It also limits all other unexpected errors to 300 characters.
